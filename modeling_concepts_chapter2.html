Introduction

Before any system is built, it helps to first understand it through a simplified representation — a model. This chapter explains what modeling is, why abstraction is central to it, and introduces the three complementary viewpoints — the class model, the state model, and the interaction model — that together give a complete picture of an object-oriented system.

2.1 Modeling

A model is a simplified representation of something, built so that it can be understood and examined before the real thing is constructed. Because a model leaves out details that are not essential to the purpose at hand, it is much easier to work with than the actual system it represents. People have used models — sketches, blueprints, scale replicas, outlines — for centuries to explore and refine ideas before committing to a final version, and software development is no different.

Models serve several distinct purposes:

Trying Things Out Early

A model lets a design be tested and evaluated before it is actually built, catching flaws while they are still cheap to fix.

Talking to Stakeholders

A model gives customers and domain experts something concrete to react to, so misunderstandings surface early rather than after the system is finished.

Seeing the Big Picture

A model helps a designer visualize how the pieces of a system fit together, revealing awkward structure before it is locked into the final product.

Taming Complexity

Perhaps the most important purpose: the human mind can only hold so much detail at once, and a model reduces a large problem to a manageable number of important ideas at a time.

2.2 Abstraction

Abstraction is the practice of deliberately focusing on the aspects of a problem that matter for a given purpose, while setting aside the aspects that do not. Because the "purpose" is what decides what counts as important, the same real-world situation can be abstracted in many different, equally valid ways depending on why the model is being built.

Every abstraction is, by nature, incomplete — no model can capture the full richness of reality, and it isn't meant to. The goal of building a model is not to arrive at some single "correct" description of the world, but to produce a description that is adequate for the job it needs to do. A model that includes unnecessary detail is not more accurate; it just makes the essential structure harder to see and prematurely locks in decisions that should have stayed open.

There is no single "right" model of a situation — only models that are more or less useful for a particular purpose. A good model keeps what matters and discards the rest.

2.3 The Three Models

No single diagram can capture everything about a system at once, so this approach describes a system from three related but distinct angles. Each model highlights one dimension of the system; together they form a complete description. Every real piece of software actually combines all three dimensions — it organizes data, it carries out steps over time, and it coordinates the work of separate parts — so a full understanding needs all three views.

The three models are not independent, self-contained slices — a system is more than the sum of separate parts — but each one can largely be studied on its own, with well-defined, limited connections to the others. As development proceeds from analysis to design to implementation, each model gains progressively more detail, though it always represents the same underlying viewpoint.

2.3.1 Class Model

The class model captures the static structure of a system: what objects exist, what identity and attributes they have, what operations they support, and how they relate to one another. It provides the foundation on which the other two models rest, since it makes little sense to talk about something changing or interacting until the "something" itself has been defined.

A good class model is expressed using vocabulary drawn from the problem domain itself — engineering terms for an engineering problem, business terms for a business problem — rather than computing jargon, at least during analysis. Class diagrams, built from classes connected by relationships such as generalization and association, are the primary notation used to express this model.

2.3.2 State Model

The state model captures the aspects of a system that depend on time and sequencing — the events that trigger change, the states an object can be in, and the rules governing how one leads to another. It describes control: the order in which things happen, independent of exactly what is being operated on or how a given step is implemented.

State diagrams are the notation used for this model. Each state diagram typically focuses on the life history of a single class of objects, showing the states it can occupy and the events that move it from one state to the next.

2.3.3 Interaction Model

The interaction model captures how separate objects work together to produce the overall behavior of a system. Where the state model looks at one class in isolation, the interaction model looks across objects, showing how they collaborate — so a complete picture of "behavior" really needs both views together.

This model is typically expressed through use cases (describing what the system does for its users at a high level), sequence diagrams (showing which objects participate in an interaction and the order of their exchanges), and activity diagrams (showing the flow of processing steps within a computation).

2.3.4 How the Models Relate

Although each model addresses a different aspect of a system, none of them stands entirely alone. The class model supplies the data that the state and interaction models act upon; operations named in the class model correspond to events and actions elsewhere. The state model shows how objects respond to conditions and change over time, while the interaction model shows how several objects combine their individual behaviors into a coherent whole.

ModelAnswersPrimary Notation
Class Model What exists, and how is it structured? Class diagrams
State Model When does something happen, and in what order? State diagrams
Interaction Model How do objects work together to get things done? Use cases, sequence diagrams, activity diagrams
Occasionally it isn't obvious which model a particular piece of information belongs to — that's expected, since any abstraction is only an approximate cut through reality. The goal is a description that is genuinely helpful, not a perfectly tidy division of every detail into exactly one box.

2.4 Chapter Summary

  • A model is a simplified stand-in for a system, built to aid understanding before the real thing is constructed.
  • Abstraction is the deliberate act of keeping what matters for a purpose and setting aside what doesn't; every model is necessarily incomplete, and that is by design.
  • A complete description of a system uses three complementary models:
    • The class model — static structure of objects and relationships.
    • The state model — how objects change and respond to events over time.
    • The interaction model — how objects collaborate to achieve overall results.
  • Different problems place different weight on the three models, but all three are needed for a full picture of a nontrivial system.
Key terms introduced in this chapter: abstraction, class model, interaction model, modeling, relationship among models, state model.
