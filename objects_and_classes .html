Objects

An object is a concept, abstraction, or thing with sharply defined boundaries and meaning for the problem at hand. Every object has three properties that together define it: identity, state, and behavior.

  • Identity — the property that distinguishes an object from all other objects, even if two objects happen to have exactly the same attribute values at some point in time.
  • State — the values held by an object's attributes at a given moment, along with its current relationships to other objects.
  • Behavior — how an object acts and reacts, expressed through the operations it can perform and the changes those operations produce in its state.

Objects can represent concrete, tangible things (a specific car, a customer, a building) as well as more abstract concepts (an event, a policy, an organization). What makes something worth modeling as an object is not its physical existence but whether it has meaningful identity, state, and behavior within the problem being studied.

Identity is what separates the object-oriented view from a purely value-based view of data — two objects can share identical attribute values and still be considered different objects because each retains its own separate identity.

Classes

A class describes a group of objects with similar attributes, common behavior, common relationships, and common semantics. An individual object is said to be an instance of its class. Defining a class does not, by itself, create any objects — it only creates a template; objects come into existence only when instances of the class are actually created.

Every object "knows" its own class, and a class packages together everything that its instances have in common:

  • The attributes that each instance carries (though each instance holds its own values for them).
  • The operations each instance supports.
  • The relationships each instance can participate in with other objects.

Classes are usually drawn as a rectangle in a class diagram, divided into up to three compartments: the class name, its attributes, and its operations.

A class box: the top compartment holds the class name, the middle holds attributes, and the bottom holds operations.

Object Diagrams

An object diagram represents the class model — it is built from classes, connected by lines showing the relationships (associations) between them. Where a class describes a whole group of similar objects, an object diagram shows the types of objects that exist in a system and how those types are related to one another, rather than depicting individual object instances directly.

Object diagrams are the primary tool used during analysis to capture the static structure of a problem: the classes involved, the attributes and operations each carries, and the associations connecting them, expressed in vocabulary drawn from the problem domain itself.

A simple object diagram: two classes, Person and Car, connected by an "owns" association with multiplicities.

Attributes

An attribute is a named data value held by every object in a class. Each object instance carries its own value for each attribute defined by its class — the class specifies what attributes exist, while each instance supplies its own value for them. An attribute's value should be a pure data value (a number, string, date, enumerated value, and so on), not a reference to another object; relationships between objects are instead represented as associations.

In a class diagram, attributes are listed in the middle compartment of the class box, one per line. An attribute name may optionally be followed by its type and a default value:

attributeName : Type = defaultValue

For example, an attribute might appear as age : Integer = 0, indicating that the attribute is named age, holds an integer, and defaults to zero unless otherwise set. How much of this detail is actually shown depends on the level of detail appropriate to the diagram — early analysis diagrams often show only attribute names, while design-level diagrams tend to include types and defaults as well.

Objects and Methods

An operation is a function or transformation that may be applied to, or by, objects in a class; all objects in a class share the same operations. A method is the specific implementation of an operation for a particular class — the actual procedure that gets executed when the operation is invoked on an object of that class.

This distinction matters because of polymorphism: the same operation, called by the same name, can have a different method — a different implementation — in different classes. Code that invokes an operation does not need to know which specific method will run; the correct method is selected automatically based on the class of the object the operation is invoked on.

Summary of Notation for Classes

ElementNotationDescription
Class Rectangle divided into up to three compartments Represents a group of objects with shared attributes, operations, and relationships
Class name Top compartment, bold, centered Uniquely identifies the class within the model
Attributes Middle compartment, one attribute per line Lists the data values each instance of the class carries
Operations Bottom compartment, one operation per line, with parentheses Lists the behavior each instance of the class supports
Attribute detail name : Type = default Optional detail showing an attribute's type and default value
Association Solid line connecting two class boxes Shows a relationship between instances of the connected classes
Multiplicity Numbers or ranges (e.g., 1, 0..*, 1..*) near a line end Indicates how many objects of one class relate to one object of the other
